# 购物车"满减凑单推荐"模块协同评审报告

> 评审时间：2026-05-19
> 参与角色：产品经理、架构专家、开发工程师
> 需求来源：运营团队
> 业务目标：提升客单价

---

## 一、需求概述

### 1.1 原始需求
在购物车页面增加"满减凑单推荐"模块，当用户购物车金额接近满减门槛时，推荐可凑单商品，引导用户加购以凑满减条件，最终提升客单价。

### 1.2 核心逻辑链
```
购物车金额 → 匹配满减规则 → 计算差额 → 推荐凑单商品 → 用户加购 → 触发满减
```

### 1.3 价值假设
- **商业价值**：客单价提升、GMV增长
- **用户价值**：帮助用户享受满减优惠、减少凑单决策成本

---

## 二、三方评审结论

### 2.1 产品经理视角

#### 需求判断
- ✅ 需求场景真实存在（用户主动凑单行为有数据支撑）
- ⚠️ 需验证：购物车页面是否是最佳推荐位置
- ⚠️ 需关注：推荐模块对购物车转化率的影响

#### 产品方案
- **推荐方案**：购物车内嵌推荐模块
- **模块位置**：商品列表下方、结算栏上方
- **展示时机**：仅当"差额 ≤ 门槛的30%"时展示
- **交互设计**：一键加购，不跳转详情页

#### 业务规则
| 规则项 | 决策 |
|--------|------|
| 推荐商品数量 | ≤6个 |
| 多档满减推荐策略 | 推荐"最近可达档" |
| 已凑满后模块状态 | 隐藏 |
| 推荐商品排序 | 按"凑单性价比"排序 |

#### 产品指标
**北极星指标**：推荐模块带来的客单价提升幅度

**过程指标**：
- 模块曝光率
- 推荐点击率
- 加购转化率
- 凑单成功率

**护栏指标**：
- 购物车转化率（不能下降）
- 购物车页面停留时长（不能显著增加）

#### AB测试策略
- 实验组：展示凑单推荐模块
- 对照组：不展示
- 实验周期：2周
- 判断标准：客单价提升 > 3% 且 转化率下降 < 1%

---

### 2.2 架构专家视角

#### 系统影响评估

**前端层面**：
- 购物车页面UI/UX需要调整，增加推荐模块区域
- 需要考虑移动端和PC端的适配
- 推荐模块的交互设计（点击、添加、移除等）

**后端层面**：
- 需要新的推荐算法服务
- 商品库存、价格、促销规则的数据集成
- 实时计算推荐结果的能力

**数据层面**：
- 用户行为数据收集（浏览、加购、购买历史）
- 商品关联关系数据
- 促销规则数据（满减、优惠券等）

#### 技术架构考虑

**推荐算法方案**：
- 基于规则的推荐：根据满减门槛计算最优凑单组合
- 基于协同过滤：根据相似用户购买行为推荐
- 基于内容的推荐：根据商品属性相似度推荐
- 混合推荐策略：结合多种算法

**性能要求**：
- 响应时间：推荐计算需要在毫秒级完成
- 并发能力：购物车页面访问量大，需要高并发支持
- 数据实时性：库存、价格、促销状态需要实时更新

#### 风险评估
| 风险类型 | 风险描述 | 缓解措施 |
|----------|----------|----------|
| 技术风险 | 推荐算法效果不佳 | A/B测试验证 |
| 性能风险 | 大促期间系统压力 | 降级策略、缓存优化 |
| 数据风险 | 库存/价格/促销状态一致性 | 实时同步机制 |

#### 实施建议
**分阶段实施**：
1. **MVP阶段**：实现基于规则的简单推荐（如"再买X元可享受Y优惠"）
2. **优化阶段**：引入个性化推荐算法
3. **扩展阶段**：增加更多推荐策略和AB测试

### 2.3 开发工程师视角

#### 技术可行性判断
**结论**：✅ 可行，但有 5 个硬约束必须前置

#### 硬约束清单

**约束1：推荐接口的响应时序**
- 购物车页是关键路径，推荐不能阻塞主流程渲染
- **方案**：异步占位，主流程先渲染，推荐模块骨架屏占位
- **硬约束**：推荐接口 P99 ≤ 300ms，超时直接降级不展示

**约束2：满减规则与推荐算法的耦合边界**
- 满减规则匹配（营销域）与凑单商品推荐（推荐域）必须解耦
- 推荐算法的输入是"差额+用户信息"，不直接依赖满减规则引擎内部状态

**约束3：推荐结果的数据一致性**
- **P0风险**：推荐了但加购后满减不生效
- **硬约束**：推荐接口返回的商品必须经过"加购模拟校验"
- **产品决策**：第一阶段可以 `verified=false` 上线，前端加提示"加购后以实际优惠为准"

**约束4：缓存策略**
- 购物车是高频变更态，不能长缓存推荐结果
- **建议**：短时窗口缓存（5-10s）+ 购物车变更事件驱动失效

**约束5：降级与兜底**
```
推荐服务超时/异常 → 模块不展示 → 购物车正常使用
推荐服务返回空结果 → 模块不展示
满减规则无匹配 → 不调推荐接口（短路）
```

#### 接口设计草案
```
GET /api/cart/bundle-recommendation

Request:
  cart_id: string          // 购物车ID
  cart_amount: number      // 当前购物车实付金额
  exclude_spu_ids: string  // 购物车内已有商品，排除

Response:
  threshold: number        // 目标满减门槛
  gap_amount: number       // 差额
  discount_amount: number  // 可享满减金额
  items: [{                // 推荐商品列表（≤6个）
    spu_id, sku_id, name, price, image, 
    add_to_cart_url        // 一键加购链接
  }]
  verified: boolean        // 是否经过加购模拟校验
```

#### 前端实现要点
| 要点 | 方案 |
|------|------|
| 模块位置 | 购物车商品列表下方、结算栏上方 |
| 渲染策略 | 骨架屏占位 → 数据到达后渐入 |
| 交互 | 单商品一键加购，加购后购物车实时刷新 |
| 状态管理 | 推荐数据独立 store slice |
| 曝光埋点 | 模块曝光 + 单商品曝光 + 加购点击 |

## 三、风险清单汇总

| 风险 | 等级 | 来源 | 缓解措施 |
|------|------|------|----------|
| 推荐商品加购后满减不生效 | **P0** | 开发 | 加购模拟校验（可分阶段实施） |
| 推荐接口拖慢购物车首屏 | P1 | 开发 | 异步占位 + 超时降级 |
| 购物车高频变更导致推荐缓存雪崩 | P1 | 开发 | 短窗口缓存 + 事件驱动失效 |
| 推荐算法与满减规则耦合 | P2 | 架构 | DTO 解耦 |
| 推荐结果与用户意图不符 | P2 | 产品 | A/B 测试验证 |
| 推荐模块影响购物车转化率 | P2 | 产品 | 护栏指标监控 |

## 六、评审结论

### 6.1 总体判断
**需求方向合理，技术上可行，但需要分阶段实施并建立完善的监控体系。**

### 6.2 关键卡点
**"加购模拟校验"是整个方案的关键卡点**——它决定了推荐的可信度，也决定了实现复杂度。

### 6.3 建议决策路径
1. 先确认待确认事项中的业务数据
2. 产品经理明确业务规则决策
3. 架构专家评估加购模拟校验的性能开销
4. 三方对齐后进入排期
